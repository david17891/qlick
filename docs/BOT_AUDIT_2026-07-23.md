# Auditoría de bots y baseline de rollback — 2026-07-23

## Baseline protegido

Antes de optimizar, el estado funcional queda identificado por el commit
`39c87f4` (`test-eventos-e2e-funnel`) en la rama `codex/e2e-funnel-test`,
correspondiente al PR #37. Ese commit contiene el smoke E2E del funnel de
eventos y no incluye los cambios visuales locales pendientes.

Para volver al baseline, se puede revertir únicamente el commit de optimización
posterior o redeployar `39c87f4` desde Vercel. No se debe usar `reset --hard`
porque el worktree contiene cambios visuales del usuario.

## Modos encontrados

La implementación tiene cinco modos configurables en `system_settings`:

| Modo | Prompt | Herramientas | Uso |
|---|---|---:|---|
| `socratic_autopilot_v2` | Socrático | Sí | fallback/default histórico |
| `socratic_no_tools_v1` | Socrático | No | contingencia sin tools |
| `super_executive` | Súper Ejecutivo V2 | Sí | modo operativo principal |
| `super_executive_v2` | Súper Ejecutivo V2 | Sí | alias/escape hatch explícito |
| `human_first` | Human-first | Sí | conversación LLM-first opt-in |

Los tres proveedores de IA (`mock`, `openrouter`, `deepseek`) son una capa
distinta: no son tres bots de conversación, sino backends intercambiables para
los cinco modos.

## Hallazgos

1. `findLeadByPhone` ya implementa timeout interno de 3 s + retry de 200 ms,
   pero `bot-engine` añadía otra carrera de 5 s. En una consulta lenta, la
   carrera externa vencía antes de que terminara el retry interno, provocando
   un falso timeout, un INSERT de fallback y ruido operativo. Esto era el
   timeout observado en el E2E.
2. El nuevo E2E se ejecutaba accidentalmente dentro de `test:ci`, aunque
   requiere Supabase y secretos explícitos. Eso hacía fallar CI en GitHub sin
   indicar un fallo de producto.
3. La selección de modos está centralizada en `pickSystemPromptForMode`; los
   cinco modos comparten el mismo provider y guardrails. No se detectó un
   segundo motor oculto que requiera una corrección separada.

## Criterio de salida

- El lookup de lead no debe tener dos timeouts superpuestos.
- `npm run test:ci` debe excluir E2E externos y seguir pasando.
- `npm run test:e2e:funnel` debe seguir pasando con WhatsApp/Brevo mock y
  webhook Stripe test firmado.
- Deben conservarse los guardrails de opt-out, consentimiento, herramientas y
  human handoff para los cinco modos.
